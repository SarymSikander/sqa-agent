# Agency Portal — Overview

> Source: Zambeel Agency Model PRD (Zambeel-Agency.pdf) + zambeel-fe source code.

## What Is It?
The Agency Portal is a sub-portal within the Seller interface, accessible to users who have registered and been approved as an Agency. Agencies are sales/growth partners that connect to multiple seller-merchants, track consolidated order performance and revenue, and earn commission on delivered orders.

## Who Uses It?
| Role | Access |
|------|--------|
| `Seller` (approved as agency) | Full Agency portal at `/agency/portal/*` |
| Agency team members | Same portal access as owner; cannot modify team |

A Seller must first register as an agency and be **approved** by an Admin before the full Agency portal unlocks.

## Registration Status Lifecycle
```
(No registration) → [Apply Now] → Pending
                                      ↓
              ┌──────────────────────────────────────┐
              ↓           ↓                ↓           ↓
          Approved     OnHold          Rejected    (stays Pending)
              │           │                │
              │     allow_resubmit=true     cooloff_days_remaining
              │           │                │
              │        Pending         (cooloff expires → reapply)
              ↓           ↓
          APPROVED    APPROVED
              ↓
          [Admin can revoke license_status = 'Revoked']
              ↓
       Revoked (shown even if registration_status = Approved)
```

> ⚠️ **Revoked overrides Approved**: If `license_status = 'Revoked'`, the Revoked card is shown regardless of `registration_status`.

## Registration Status Values
| Status | Who Sets | What User Sees |
|--------|----------|----------------|
| `null` / none | — | Agency Program landing page with "Apply Now" CTA |
| `Pending` | Auto after register | Animated clock, "Application Under Review", agency name + submission date |
| `Approved` | Admin approves | Green checkmark, "Approved!", Agency ID with copy, "Open Agency Portal" button |
| `OnHold` | Admin puts on hold | Amber alert, hold_reason text, optional Resubmit CTA |
| `Rejected` | Admin rejects | Red X, reject_reason, cooloff_days_remaining |
| `Revoked` (via license_status) | Admin revokes | Red X, "Agency Access Revoked", contact support link |

## Business Rules
- **One agency per user**: A user can only have one agency registration. Duplicate registrations are rejected by the backend.
- **Country locking**: Country cannot be changed after registration — commission models are country-specific.
- **Cooloff period**: Rejected agencies have `cooloff_days_remaining` — reapplication allowed only after cooloff expires.
- **Terms enforcement**: `terms_accepted` is validated both frontend and backend.

## Post-Login Landing for Approved Agency
After login, approved agencies land on `/get-started` or `/dashboard` (Seller portal). The Agency Portal is accessed by clicking the **Agency** tab in the sidebar.

## Route Guard
- Pages at `/agency/*` — protected by `SellerProtectedRoute`
- Pages at `/agency/portal/*` — additionally require `AgencyApprovedRoute` (registration_status === "Approved" AND license_status ≠ "Revoked")
- If not approved, `/agency/portal/*` redirects to `/agency`

## Agency Unique ID Format
- Format: `ZMB-AG-XXXXXX` (6 alphanumeric chars after prefix)
- Generated by the backend **at approval time** — NOT at registration time
- IDs are **immutable** — cannot be changed or reassigned
- Merchants validate client-side: must start with `ZMB-AG-`
- Merchants use this ID to connect to the agency via `POST /api/merchant/agency/connect`

## Commission Types
| Type | How Calculated |
|------|---------------|
| `percentage_of_delivered_revenue` | `commission_amount = delivered_revenue × (value / 100)` |
| `flat_per_delivered_order` | `commission_amount = value × delivered_orders` |

- Commission is calculated only on **delivered** orders.
- Rule matched by `order_country` of the merchant's order.
- If no rule exists for that country, no commission is recorded.
- Commission records are **immutable** once created (audit trail).

## Invoice Statuses
`Unpaid` → `Paid` (Admin updates when payment is made)

## Merchant Connection Statuses
| Status | Display | Meaning |
|--------|---------|---------|
| `Pending` | Pending | Merchant sent request, not yet accepted |
| `Active` | Active | Agency accepted — connection live |
| `Rejected` | Rejected | Agency rejected the request |
| `Disconnected` | Inactive | Merchant or revocation disconnected |

- **One agency per merchant at a time**: A merchant can only be connected to one agency.
- If agency's license is revoked, all connections move to `Disconnected` automatically.
- `disconnected_at` and `disconnect_reason` are permanently recorded on the connection record.

## Team Member Roles
- `owner` → displayed as `Admin`; cannot be removed; only one per agency
- `member` → displayed as-is; same portal access as owner except cannot modify team

## Team Member Statuses
- `Active` — accepted the invite
- `Invite Pending` — invited but not yet accepted (`invite_accepted = false`)

## Access Scopes (when merchant connects)
- `"all"` — agency sees all merchant stores
- `"specific"` — agency sees only specified store IDs

## Contextual Store Switching
- Agency clicks "Open →" on a store in Merchant Detail Drawer
- `AgencyStoreContext` written to Zustand (`useAgencyViewStore`) with: merchantId, merchantName, merchantEmail, storeId, storeName, storeCountry, agencyName
- User navigated to `/dashboard` (merchant's seller dashboard)
- Agency Context Banner shown on every page: **Agency Name · Merchant Name · Store Name**
- Context cleared when navigating to `/agency/portal/*`
- Backend validates active connection on every contextual API call — returns 403 if disconnected
